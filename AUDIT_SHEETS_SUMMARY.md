# Sheets and cells audit summary

This document summarizes expected sheet structure and critical constraints so all code and formulas read/write correctly.

## Admin_Profiles

- **Sheet name:** `Admin_Profiles`
- **Row 1:** Headers (profileId, displayName, email, status, statusReason, isAdmin, ...). Column order is determined by header names; scripts use `headers.indexOf("isAdmin")` etc.
- **Data:** Row 2 onward.
- **Critical columns:**
  - **status:** Must be `active` or `inactive_soft_locked` (data validation). Used by QUERYs in Admin_Analytics (e.g. `Admin_Profiles!A:D` where D is status).
  - **isAdmin:** Must be the string `TRUE` or `FALSE` only. Data validation is set to `["TRUE", "FALSE"]`. Any other value (e.g. "true", "Yes", 1, empty) triggers a validation error (e.g. "cell AA2 violates the data validation rules"). All script writers use `"TRUE"` or `"FALSE"`: admin_api.js (adminSetIsAdmin, adminUpdateProfile), sheet_format.js (ensureAdminProfilesView_), admin_profiles_onedit.js (normalizes on edit).
- **Writers:** profiles.js (loadProfiles_ read-only), admin_api.js (adminUpdateProfile, adminSetIsAdmin), skill_profile_writeback.js, resume_staging.js (apply lanes writes roleTracksJSON). isAdmin is written only via admin_api or onEdit; both use canonical "TRUE"/"FALSE".

## Resume_Staging

- **Sheet name:** `Resume_Staging`
- **Headers (row 1):** profileId, roleTitle, keywords, confidence, reason, Approved, Applied (7 columns).
- **Data:** Row 2 onward. resume_staging.js writes with 7 columns; Approved is boolean/TRUE-FALSE compatible.
- **Used by:** resume_staging.js (writeStagingFromParsed_, getAllStagingRowsForProfile_, applyApprovedLanes_), sheet_format.js (formatResumeStagingSheet_ if present). Admin_Analytics may reference; confirm QUERY column letters if used.

## Engine_Inbox / Engine_Tracker

- **Sheet names:** `Engine_Inbox`, `Engine_Tracker`
- **Profile-scoped:** Rows filtered by profileId. admin_analytics.js QUERYs assume specific columns (e.g. Engine_Inbox C = tier, G, J = laneLabel; Engine_Tracker H = status, K = laneLabel). engine_tables.js or equivalent defines column layout; QUERYs in admin_analytics must match.

## Logs

- **Sheet name:** `📓 Logs`
- **Row 1:** Filter row (dropdowns). Row 2: headers. Row 3+: data.
- **Columns:** A–G (time, profile, action, source, details, etc.). admin_analytics QUERYs use `'📓 Logs'!A2:G` and columns D (action), G (level). logging.js and sheet_format (formatLogsSheet) must append with same column count and order.

## Admin_Analytics

- **Sheet name:** `📊 Admin_Analytics` or `Admin_Analytics` (confirm in admin_analytics.js).
- **Content:** Generated by admin_analytics.js. QUERYs reference: Admin_Profiles!A:D (D = status), '📓 Logs'!A2:G, Engine_Inbox, Engine_Tracker. If Admin_Profiles column order changes so status is not D, QUERYs must be updated.

## Audit checklist (verified)

- **Resume_Staging:** 7 columns; resume_staging.js and format logic aligned.

- **Admin_Analytics QUERYs:** Assume Admin_Profiles D = status; Logs A2:G; Engine_* columns as in admin_analytics.js. No code changes in this audit; document for future column renames.
